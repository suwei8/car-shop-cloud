# 远程开发实施交接清单

本文件用于后续交给远程开发服务器上的 agent 执行开发。当前阶段不连接远程服务器、不初始化代码。

## 1. 项目目标

开发一套 SaaS 模式的汽车行业服务管理系统，参考百易云修的业务覆盖，但先以可落地 MVP 为目标。系统必须支持多商户租户隔离和多门店，商户按套餐添加门店和员工；第一版提供平台运营后台、商户 Web 管理后台和安卓 APP；看板作为 Web 大屏能力建设。

## 2. 推荐技术架构

如果没有既定技术栈，建议：

- Web 前端：React + TypeScript + Vite + Ant Design Pro 或 React Router + TanStack Query。
- 安卓 APP：uni-app、Flutter 或 React Native。若优先开发速度和后续多端复用，建议 uni-app；若追求长期原生体验，可选 Flutter。
- 后端：NestJS 或 Spring Boot。若团队偏 JS，全栈统一使用 NestJS。
- 数据库：PostgreSQL。
- 缓存/队列：Redis，二期用于消息、导出、报表异步任务。
- 文件存储：S3 兼容对象存储，用于车辆照片、施工照片、附件。
- 鉴权：JWT + Refresh Token，服务端保存会话版本。
- 权限：平台权限 + 商户 RBAC + 数据范围。
- 部署：Docker Compose 起步，后续 Kubernetes。

## 6. 开发原则

- 所有业务表带 `tenant_id`，门店相关数据带 `shop_id`。
- 所有业务查询默认必须带租户作用域，禁止依赖前端传入的 `tenant_id` 决定数据边界。
- 平台后台和商户后台使用不同权限命名空间，避免平台能力泄漏给商户用户。
- SaaS 套餐和功能开关必须在后端校验，前端隐藏菜单不能作为权限依据。
- 金额使用整数分或 decimal，禁止 float。
- 工单、库存、结算必须有操作日志。
- 第一期库存固定为工单确认施工/出库时直接扣库存，不等结算，不实现仓库领料审核流程。
- 所有库存变动必须写入 `stock_movements`，记录来源单据、来源明细、操作人和变动后余额。
- 删除优先软删除，财务和库存单据不允许物理删除。
- 已结算工单只能通过反结算或调整单处理。
- 手机号、车牌、VIN 后续要支持脱敏展示。
- 列表页从第一天就支持分页、筛选、排序。
- 报表查询不要直接依赖前端拼装，应提供后端聚合接口。
- Web 和安卓 APP 必须共享同一套后端权限和业务状态机。
- 储值卡第一期必做，余额必须区分本金和赠送金额。
- 储值卡消费必须同时写入结算支付明细和储值流水，禁止只改余额。
- 储值卡退款、作废、停用必须记录原因和操作日志。
- 套餐卡第一期必做，必须按服务项目记录总次数和剩余次数。
- 套餐卡核销必须同时写入工单项目抵扣明细和套餐流水，禁止只改剩余次数。
- 套餐卡必须校验适用车辆、适用门店、有效期、剩余次数和卡状态；前端提示不能替代后端校验。
- 套餐卡退款、作废、停用必须记录原因和操作日志。
- 安卓 APP 员工登录后只能访问其商户和授权门店的数据。
- 文件上传必须记录来源端、业务对象、上传人和租户。
- 第一期只记录收款方式和流水号，不接入微信/支付宝真实支付。
- 第一期不做历史数据迁移。
- A4 工单和 A4 结算单打印一期必做。

## 7. 风险与待定事项

- SaaS 套餐如何定价和分级。
- 试用期、到期停用、续费提醒规则。
- 平台运营人员是否允许代登录商户后台。
- 安卓 APP 是否需要上架应用市场，还是先 APK 分发。
- 是否规划微信小程序作为二期。
- 是否需要对接第三方 VIN/EPC/保养数据。
- 是否需要热敏小票打印。
- 是否要求离线使用或弱网缓存。

## 8. 远程服务器交接注意

远程服务器凭证由用户提供，但当前调研阶段不使用。正式进入开发阶段时，再在远程服务器：

1. 检查系统环境。
2. 初始化 Git 仓库或拉取已有仓库。
3. 创建 `.env.example`，不提交真实密钥。
4. 建立数据库迁移。
5. 按 Sprint 0 开始实施。
