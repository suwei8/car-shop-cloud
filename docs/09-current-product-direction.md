# 当前产品方向与核心闭环

> 状态：当前权威产品方向
> 更新时间：2026-06-16
> 适用范围：后续任务书、需求评审、Agent 派发与验收

## 1. 产品定位

车店云管家面向 **1-5 人小型汽车服务门店**，包括夫妻汽修店、快保店、洗美店、轮胎/保养店和小型综合汽服店。

产品不追求覆盖大型汽修厂、连锁总部、复杂财务、人事绩效、保险理赔和大型 ERP 场景。大型汽修厂通常有更高预算和成熟系统选择，本项目的机会在于为小店提供：

- 买得起；
- 手机上能用；
- 不需要专门培训；
- 直接改善日常开单、收款、库存、客户留存和老板看账。

## 2. 端形态决策

第一优先级是 **商户端微信小程序**，供门店老板和员工内部使用。

- 小程序：日常 90% 高频操作，注册开户、开单、收款记录、客户车辆、配件库存、储值/次卡、完工通知、老板看账、订阅续费。
- Web 后台：重操作辅助端，批量导入、复杂报表、打印模板、权限配置、平台运营管理。
- Android/iOS App：后续可适配，不作为当前第一主线。
- 车主端/车主小程序：当前不做，不作为套餐卖点。车主触达仅保留短信/通知能力。

## 3. 当前核心闭环

当前核心闭环应收口为：

```text
老板注册开通试用
  → 建立客户/车辆
  → 快速开单（维修/洗美/快保）
  → 选择服务与配件
  → 完工时扣库存并沉淀质保记录
  → 记录收款/欠款/储值卡或次卡核销
  → 老板看今日收入、欠款、库存预警、客户提醒
  → 试用到期后小程序内续费
```

其中最重要的价值点是：

1. **快速开单和收款记录**：让小店从纸本、微信聊天、记账本迁移到系统。
2. **客户车辆历史**：下次到店能快速知道上次做了什么、花了多少钱。
3. **配件库存**：小店也会因为机油、滤芯、轮胎、火花塞等常用件库存不准而损失钱。
4. **质保追溯**：能回答“上次换的件还保不保”，这是小店建立信任的低成本能力。
5. **储值卡/次卡**：对洗美、快保、保养套餐是强商业化抓手。
6. **老板看账**：今天收了多少钱、还有多少欠款、哪些配件快没了。
7. **试用到续费**：产品自身要形成 SaaS 收费闭环。

## 4. 应当弱化或延后的能力

以下能力不应进入当前主路径：

- 多级总部、复杂连锁管控；
- 复杂派工、班组、工位、绩效；
- 复杂仓储、领料审批、调拨、云仓；
- 车主小程序、积分商城、社群营销；
- 保险、理赔、事故车全流程；
- 工资、社保、复杂财务封账；
- EPC/保养手册自建数据库；
- 门店向车主的真实在线支付闭环。

这些能力可以保留扩展空间，但不应成为当前小店版默认体验。

## 5. 可以加入但应轻量化的能力

以下能力和目标客户匹配，但必须保持简单：

- 欠款提醒：记录谁欠多少钱，老板能一键查看。
- 保养/年审/客户流失提醒：先做门店内部待办，不直接做营销轰炸。
- 小票打印：优先云小票打印机，A4 留给 Web 后台。
- OCR/VIN：作为开单辅助，失败不能阻塞开单。
- 员工邀请：以老板邀请员工加入为主，不暴露完整 RBAC 复杂度。
- 常用项目/配件模板：按汽修保养、洗美快修、综合汽服预置。

## 6. 外部调研摘要

外部资料与竞品共同指向一个结论：小型维修门店最需要的不是“大而全 ERP”，而是把预约/开单、客户车辆、配件、结算、提醒和经营数据放到一个简单流程中。

- Square 面向汽车服务门店的方案强调收款、预约、客户回访、现金流和经营表现跟踪。
- Orderry 将工单、配件、客户、付款、短信提醒和维修状态集中到一个工作流。
- McKinsey 关于汽车后市场的研究强调数字渠道、客户期望变化和独立维修渠道的重要性。
- NAPA TRACS 等维修门店软件强调排程、开票、支付和减少纸面管理。

## 7. 技术架构原则

- 后端仍按 SaaS 多租户设计，所有业务数据必须租户隔离。
- 开户主要来自小程序自助注册：老板打开小程序，手机号验证码注册，自动创建租户、默认门店、默认仓库、管理员账号、预置服务项目和试用订阅。
- 前端不能提交或决定 `tenantId`，后端必须从 JWT 推导租户。
- 小程序优先，Web 为辅；共享 API，不做车主端 API。
- 支付分两类：
  - 门店向车主收款：当前只记录现金/微信/支付宝/储值卡/挂账，不接真实支付。
  - 商户向平台买套餐：可以接微信支付，用于 SaaS 订阅续费。
- OCR/VIN、短信、营销都必须是可降级能力，不能阻断开单和结算主链路。

## 8. 后续任务书编写规则

后续任务书必须先判断是否服务于上述核心闭环。若不服务于小店日常经营或 SaaS 收费闭环，应默认后置。

每个任务书必须写清：

- 目标用户是谁；
- 解决哪一个小店业务问题；
- 是否影响租户隔离、金额、库存、结算、储值卡、套餐卡；
- 必填验证命令；
- Agent 回执区域；
- 架构师审核区域。
