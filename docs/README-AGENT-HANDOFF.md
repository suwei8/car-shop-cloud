# 车店云管家开发交接说明

本目录是「车店云管家」当前版本开发的产品与架构交接包。远程开发 Agent 开始实现前，请先按顺序阅读本文件列出的文档。

## 1. 当前最终决策

- 产品名暂定：车店云管家。
- 商业模式：SaaS 年费套餐 + 增值服务。
- 当前主入口：商户端微信小程序，供门店老板和员工内部使用。
- Web 后台作为重操作辅助端：导入、报表、打印、权限、平台运营。
- Android/iOS App 后续可适配，不作为当前第一主线。
- 当前不做车主端/车主小程序，不作为套餐卖点。
- 第一版不做保险业务。
- 门店向车主收款：第一版只记录收款方式、金额和流水号，不接真实门店支付。
- 商户向平台购买套餐：允许接微信支付，用于 SaaS 订阅续费。
- 重点服务 1-5 人小型汽修/洗美/快保门店，默认小店轻量体验。
- 系统保留多商户、多门店、多员工能力，但默认体验优先单店/少员工。
- 第一版支持维修工单 + 洗美快单。
- 第一版支持轻量健康检查。
- 第一版支持库存，扣减时点为工单确认施工/出库时直接扣库存，不等结算。
- 第一版支持储值卡和套餐卡。
- 套餐卡必须限制适用车辆、适用门店、有效期。
- 小程序优先支持小票/收费清单，A4 工单和 A4 结算单主要放在 Web 后台。

## 3. 当前任务组织

历史任务分为：

- `docs/tasks/`：基础业务任务，已完成。
- `docs/tasks-phase2/`：SaaS 商业化基建任务，主体已完成；其中车主端任务为历史方向，当前已下线，不再作为产品主线。
- `docs/tasks-phase3/`：上线、支付、安全、分析、营销等任务，README 以最新同步后的状态为准。

后续新任务默认以 `09-current-product-direction.md` 为产品判断依据。

## 4. 开发硬约束

- 所有业务表必须包含 `tenant_id`。
- 门店级业务数据必须包含 `shop_id`。
- 后端必须强制租户隔离，不能依赖前端传入 `tenant_id`。
- 套餐和功能开关必须后端校验。
- 金额不能使用 float。
- 库存、结算、储值卡、套餐卡必须写流水和审计日志。
- 已结算工单不能直接修改核心金额，应通过反结算或调整流程处理。
