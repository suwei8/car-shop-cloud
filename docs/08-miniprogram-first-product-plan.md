## 0. 本方案要解决的问题

把产品从“Web 后台为主的泛汽服 SaaS”，调整为：

> **以微信小程序「车店管家」为主入口、面向 1-5 人小型汽修/洗美店的轻量 SaaS，老板用手机即可完成注册、开单、收款、库存、看账；Web 后台作为重操作的辅助端。**

注意区分：

- 本方案的小程序是 **商户端**（给门店老板/员工用），不是车主端。
- 车主小程序仍然 **不做**（与 `TASK-PM-001` 一致）。

## 3. 产品定位

一句话定位：

> 车店管家是一套专为 1-5 人汽修/洗美小店设计的轻量管理小程序，手机上就能开单、收款、管客户、管配件、查质保、看收入。

目标客户：

- 夫妻汽修店、保养店（1-3 人）；
- 小型综合汽服店（3-5 人）；
- 洗美/快修小店（高频、套餐卡需求强）。

不主攻：大型综合修理厂、连锁总部（架构保留能力，但不作销售主线）。

## 4. 端的分工（核心原则）

> **小程序为主，Web 为辅，不是二选一。**

### 4.1 小程序（主入口，日常 90% 操作）

- 注册开户与试用开通；
- 老板首页（看账）；
- 快速开单（维修单 / 洗美快单）；
- 收款 / 欠款记录；
- 客户车辆检索与历史；
- 配件库存查询（含低库存、质保、供应商电话）；
- 储值卡 / 套餐卡售卡与核销；
- 完工与通知；
- 套餐购买与续费（微信支付）。

### 4.2 Web 后台（辅助，重操作）

- A4 工单/结算单打印与打印模板；
- 批量库存管理、入库、盘点；
- Excel 数据导入；
- 报表查看与导出；
- 员工/角色/权限配置；
- 系统参数、字典、编码规则。

### 4.3 为什么这样分

- 小程序打印能力受限（A4 基本做不了，重表格体验差）；
- 批量/配置类操作在手机上效率低；
- 但日常高频动作（开单、收款、看账）在手机上反而最顺。

### 4.4 打印方案（已确认）

- 最终对接 **云小票打印机**（小票尺寸，适合小店）；
- 店内收款用 **收款码或现金**，不强求系统内真实支付；
- 小程序可将 **收费清单打印出来给客户**（小票/收费明细），不强求 A4；
- A4 工单/结算单等正式单据，留在 Web 后台按需打印。

## 5. 注册开户流程（优化版）

### 5.1 设计原则

1. 先让老板用起来，再谈钱：**注册即开通试用，不在注册阶段强制选套餐/付费**。
2. 注册步骤尽量短，目标 **30 秒内进入首页**。
3. 用经营类型 + 员工数自动配置默认体验（简易模式、预置项目）。

### 5.2 推荐流程

```text
1. 打开小程序「车店管家」
2. 微信登录（拿 openid 做免密登录）
3. 输入手机号 → 短信验证码校验（自有短信通道）
4. 选择经营类型（汽修 / 洗美快修 / 综合汽服）
5. 填写店名（必填）、员工数（必填）、地址/联系电话（选填）
        ↓ 员工数 ≤ 5 默认开启「简易模式」
6. 自动开通 30 天免费试用 + 预置常用服务项目 + 默认仓库
7. 直接进入首页，引导「开第一张单」
8. 试用期内或点击付费功能时，再引导「选择套餐 → 微信支付」
```

### 5.3 与原构想的差异

| 项 | 原构想 | 调整后 | 原因 |
|----|--------|--------|------|
| 选套餐时机 | 注册时强制选 | 后置到试用期内/用付费功能时 | 降低注册流失，先体验后付费 |
| 免费档 | 基础永久免费版 | 30 天试用（默认不永久免费） | B 端永久免费支持成本高、转化低 |
| 手机号获取 | （未定） | 手机号 + 短信验证码 | 不采用按次计费的微信一键授权 |
| 账号 | — | 注册人=老板=租户管理员，自动初始化 | 复用已有 seed |

### 5.4 经营类型如何影响初始化

- 汽修/保养店：预置保养、机修类常用服务项目与常用配件分类；
- 洗美快修店：预置洗车、美容、精品类项目，开单路径更短；
- 综合汽服店：两者合并，默认仍开简易模式。

## 6. 小程序核心 5 屏（首期范围）

### 6.1 首页（老板看账）

展示：

- 今日收入、今日工单数；
- 客户欠款总额（**当前缺口，需补**）；
- 低库存预警数；
- 待办提醒（保养到期 / 客户流失）；
- 快捷入口：开维修单、开洗车单、查客户、收欠款、会员充值。

### 6.2 开单

- 车牌/手机号快速检索老客户，自动带出车辆；
- 选服务项目 + 配件；
- 金额自动计算；
- 简易模式下「一键完工」。

### 6.3 客户

- 车牌/手机号/VIN 搜索；
- 客户车辆档案、维修历史、消费历史；
- 储值卡 / 套餐卡余额；
- **质保查询：上次换的件保到什么时候（当前缺口，需补）**。

### 6.4 配件库存（当前移动端缺失，需新增）

- 配件搜索、当前库存；
- 低库存高亮；
- 质保月数、供应商及电话；
- 简单入库（选配件、数量、进价、供应商、质保）。

### 6.5 我的店

- 门店信息、员工管理（见第 6.7 节）；
- 套餐与试用状态、续费入口；
- 跳转 Web 后台的引导（打印、报表、导入等重操作）。

### 6.6 储值卡 / 次卡（保留但简化）

定位：储值卡和次卡是小店（尤其洗车/洗美/快保）**最强付费理由之一**，保留，但在小程序里做减法。

系统内为两类卡（无独立会员等级体系）：

- 储值卡：充钱送钱，按**金额**抵扣（`StoredValueCard`）；
- 次卡 / 套餐卡：按**次数**核销，如洗车 10 次卡、保养套餐（`PackageCard`）。

简化原则：

- 简易模式默认只保留：开卡 / 充值 / 赠送 / 消费扣减 / 查余额 / 退卡；
- 高级项（适用车型、适用门店、复杂有效期规则）默认隐藏，专业版再放开；
- **不做** 会员等级、积分、折扣体系（对小店过重，后置）；
- 命名面向老板：储值卡 / 会员储值、次卡 / 套餐卡，避免术语化。

业态差异：洗车/洗美/快保为核心刚需；纯机修/事故店价值中等，仍保留。

### 6.7 员工管理（按需，入口在「我的店」）

> 定位：员工管理**不是注册时的必经步骤**，而是「我的店」里的一个入口，老板有需要时（如要加技师/前台）再点击。很多夫妻店可能从不使用，这是正常的。

角色简化（不向小店暴露完整 RBAC）：

| 角色 | 权限 |
|------|------|
| 老板 / 管理员 | 全部，含看账、设置、加人 |
| 员工（技师/前台） | 开单、收款、查客户、做工单；看不到经营报表、欠款总额、成本价等敏感信息 |

> 简易模式下可只给「老板 / 员工」两种；更细的角色（技师只看自己派工、前台管接待）留待专业版。

**主方式：邀请二维码**

```text
1. 老板在「我的店 → 员工管理 → 添加员工」
2. 选角色，系统生成有时效的邀请二维码/链接
3. 员工微信扫码打开小程序 → 微信登录 → 手机号 + 短信验证码
4. 自动加入门店、分配预设角色，立即可用
5. 老板在列表可改角色 / 停用
```

**备选方式：按手机号添加**

```text
老板填 姓名 + 手机号 + 角色 → 保存
员工首次用该手机号 + 短信验证码登录即激活
```

关键约束：

- 添加时校验套餐员工数配额（单店版 5 人），超额引导升级；
- 离职用「停用」而非删除，历史工单/收款归属不可丢（符合硬约束）；
- 员工默认无法查看经营报表、欠款、成本价（复用 `applyDataScope`）；
- 第一个注册人=老板，不可被普通员工停用或降权；
- 注册时填的「员工数」仅用于决定默认配置，**不强制在注册阶段创建员工账号**。

## 7. 商业化与套餐

### 7.1 套餐结构（建议，最终以定价文档为准）

| 套餐 | 适用 | 备注 |
|------|------|------|
| 单店版 | 1-5 人单店（主力客户） | 主打低门槛，开单/收款/客户/库存/质保/会员卡/看账 |
| 专业版 | 更成熟/综合汽服门店 | 在单店版基础上增加：派工、技师产值、完整库存（盘点/成本/多仓库）、高级报表与导出、营销提醒、更多打印模板 |
| 连锁版 | 多门店 | 多门店看板、跨店客户/储值套餐；不作销售主线，定制/咨询为主 |

价格区间见 `05-naming-and-pricing.md`（建议单店版首年 1280-1980 元或月付）。

> 套餐档位已定（三档）：**单店版 → 专业版 → 连锁版**。「单店版」为主力入门档，面向占比最多的单店客户，命名中性专业，不含贬义。

### 7.2 免费/试用策略（默认方案，待最终拍板）

- 默认：注册即开通 30 天全功能试用；
- 到期前 7 天小程序内提醒续费；
- 到期后进入受限只读/锁定（保留数据，引导付费）；
- 不做永久免费档。

> 备选：若坚持要免费档，则做“极简永久免费档”（1 人、仅开单收款、不含库存/报表/会员卡、限客户数），仅作获客漏斗。

### 7.3 支付范围（第一版）

- 收取：平台 SaaS 套餐费（小程序内微信支付）；
- 不收取：门店向车主的真实收款（仅记录方式与流水号）。

## 8. 合规与风险提醒

- 平台只需注册 **一个** 商户端小程序，多租户共用（已有 appid）。
- 小程序需配置：相应经营类目、隐私协议、用户信息授权说明。
- 微信支付（收套餐费）需平台主体企业资质与微信支付商户号。
- 短信验证码走自有通道，注意限流与一次性消费（已有基础）。
- 不采用微信“手机号快速验证（getPhoneNumber）”，避免按次计费成本。

## 11. 决策记录

全部已确认：

- 免费策略：30 天免费试用，不做永久免费；
- 主入口：小程序为主 + Web 为辅（打印/报表/导入/权限留 Web）；
- 门店收款：第一版只记录方式；SaaS 套餐费用小程序微信支付；
- 打印：对接云小票打印机，小程序打印收费清单给客户，不强求 A4；
- 注册：手机号 + 短信验证码（不采用微信一键手机号授权）；
- 储值卡/次卡：保留但简化，不做会员等级；
- 员工管理：邀请二维码为主 + 手机号添加为辅，角色简化；**按需使用**，入口在「我的店」，不在注册流程中强制；
- 套餐档位（三档）：单店版 → 专业版 → 连锁版（「单店版」为主力入门档）；
- **第一版小程序范围锁定**：注册 + 核心 5 屏（首页/开单/客户/库存/我的店）+ 套餐支付。员工邀请作为「我的店」内的按需入口随核心屏一并交付，但不单列为里程碑。

下一步：将本方案拆成可派发的执行任务书（onboarding 后端扩展、小程序 onboarding、核心 5 屏、单店版缺口补齐、套餐支付），并配好派发词与回执区。
